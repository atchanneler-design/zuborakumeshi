「自炊の意思決定コストをゼロにする」という要件を満たしつつ、個人開発としてのランニングコスト（維持費）を抑え、高速にリリースするための設計案です。開発作業自体はClaude Codeが行う想定で、本設計はその土台となるガイドラインです。

## 1. システム・アーキテクチャ全体像

ズボラキッチンは、スマートフォンから手軽にアクセスできるWebアプリ（ブラウザで動くアプリ）として構築します。

*   **フロントエンド（ユーザーが見る画面）**: ユーザーの操作やカメラ撮影を受け付けます。
*   **バックエンド（中継役・頭脳）**: 画面から受け取った画像をAI（Claude）に渡したり、生成されたレシピを画面に返したりします。
*   **AI（Claude）**: 「画像から食材を読み取る目」と「在庫からレシピを考える頭脳」の2つの役割を担います。

## 2. 技術選定と代替案

企画書への賛同をベースに、MVP向けの最適な技術を選定しました。どのような技術を使っても実現は可能ですが、運用コストと開発スピードを重視しています。

### ① フレームワーク・ホスティング（土台と置き場所）
*   **推奨案**: **Next.js (App Router) + Vercel**
    *   *説明*: Webサイトの画面と裏側の処理をセットで作れる枠組みと、それをインターネット上に公開するためのサービスの定番です。「チェッカーシリーズ」等で使い慣れているため、開発トラブルが起きにくく最速です。

### ② 画像認識・レシピ生成AI
*   **推奨案**: **最新のマルチモーダルAI（Claude / GPT-4o 等）**
    *   *画像認識（目）*: 画像と文字の両方を理解できるAI（最新の **Claude Sonnet** や **GPT-4o** など）を使用します。精度の高いモデルを「目」として使います。
    *   *レシピ生成（脳）*: テキストのみを扱う「頭脳」は、圧倒的に利用料が安い軽量モデル（Claude Haiku や GPT-4o-mini 等）に任せて維持コストを抑えます。
    *   *他で認識する方法（代替案の検討）*:
        1.  **Google Cloud Vision API 等の専用システム**: レシート（文字）だけを読むなら非常に優秀ですが、冷蔵庫の中身（物体）は認識できません。今回は冷蔵庫・レシート両方に対応したいため、AI（Claude等）に軍配が上がります。
        2.  **専用の画像認識モデル（YOLO等）**: 自動運転などで使われるシステムです。「これがニンジン」「これが豚肉」と大量に学習させる期間が必要になるため、初めから物知りなAIを利用する方が、個人開発として最速・最安です。

### ③ 状態管理・データベース（記憶の仕組み）
*   **推奨案（MVP向け）**: **Zustand + LocalStorage**
    *   *説明*: ユーザーのスマホのブラウザ内に、現在「冷蔵庫に何が入っているか」を一時記憶させる仕組みです。サーバーにデータを保存しないため維持費がゼロです。
*   *他の方法（フェーズ2以降向け）*: **Supabase**
    *   *説明*: ユーザー登録や、スマホを変えてもデータが引き継げる仕組みが必要になった場合は、設定が簡単でVercelと相性が良いSupabase（データベースサービス）を追加するのがおすすめです。

## 3. アプリの画面構成と導線（ルーティング）

アプリ全体の画面の流れ（URL構造）です。

1.  **トップ画面 (`/`)**
    *   サービスの説明と、「冷蔵庫を撮る」「レシートを撮る」の2つのボタンを配置。
2.  **食材スキャン・編集画面 (`/fridge`)**
    *   撮影後、AIが読み取った結果（例：「にんじん 1本」等）をリストで表示。
    *   ユーザーが手動で個数を修正したり、追加・削除ができます。
    *   「この食材でレシピを考える」ボタンで次へ。
3.  **レシピ提案画面 (`/recipes`)**
    *   以下の4つの切り口でレシピを並べて表示します。（※全て「買い出し不要」が前提です）
        *   **【爆速】**: ワンパン・レンチンなど10分以内でできる最短ルート
        *   **【定番】**: 奇をてらわない、誰でも知っているお馴染みの安心の味
        *   **【一掃】**: 期限の近い食材や端切れを優先的に使い切る（冷蔵庫リセット）
            *   *判定方法*: ユーザーが食材リストで「使い切りたいマーク」を付けたものを最優先しつつ、AIの知識で「傷みやすいもの（もやし、ひき肉等）」を自動で優先消費します。
        *   **【二刀流】**: 夜ご飯＋「明日の弁当」や「下味冷凍」を同時に作る効率重視レシピ
    *   各レシピの簡単な手順と、「クラシル等への検索リンク」を表示。
    *   「これを作った！」ボタンを押すと、使った食材が冷蔵庫の在庫から自動で減り、トップ画面に戻ります。

## 4. コスト・効率化の設計（非常に重要）

AIの利用料金（トークン消費）や処理時間を最小限に抑えるため、以下の工夫を設計に組み込みます。これらは実装時にClaude Codeへ厳格に指示をします。

1.  **クライアント側での画像リサイズ**
    *   スマホの写真は非常に大きく、そのままAIに視せると「視覚トークン（見るための費用）」が跳ね上がります。アプリ内で自動的に横幅1000px程度に縮小してから送信することで、1回あたりのコストを数円単位まで下げます。
2.  **「一掃」のロジックをプロンプトで制御**
    *   レシピ生成時に AI へ「ひき肉、葉物野菜、もやしなど、傷みやすい食材を優先的に使い切るメニューを考案せよ」と指示を組み込むことで、写真から日付が読めなくても「一般的に見てヤバイもの」から消費させます。
3.  **プロンプト（AIへの指示）の軽量化とJSON出力**
    *   レシピ生成を依頼する際、AIの出力は「JSON（プログラムが読みやすいデータ形式）」のみに限定します。「承知いたしました。以下のレシピを〜」といった余計な挨拶文の出力を禁止し、文字数を削ってコストを下げます。
3.  **ループ内でのAPI呼び出し禁止**
    *   レシピを5つ提案する際、1つずつ5回AIに相談するのではなく、「5つのレシピを1回の相談で全部出してください」と指示を一括化し、無駄な通信と時間を防ぎます。

## 5. APIエンドポイント設計（見えない通信作業）

画面とAIの仲介役として、以下の入り口（API）を用意します。

*   `POST /api/vision/parse-fridge`
    *   役割: スマホから画像を受け取り、Claudeに投げて「食材リスト」をプログラムデータとして返す。
*   `POST /api/recipe/generate`
    *   役割: 現在の食材リストを受け取り、Claudeに投げて「5種類のレシピセット」をプログラムデータとして返す。
*   `GET /api/og`
    *   役割: SNS（Xなど）でシェアされた時に表示される「今日のズボラ飯！」のような宣伝画像を、自動で生成する機能。

## 6. 開発フェーズと検証の流れ（Claude Codeによる実装手順）

いきなり全てを作らず、最小の動作確認（スモークテスト）を重ねながら完成させます。

*   **Step 1. 基盤構築**: Next.jsプロジェクトの作成と、見た目を作るツールの導入。
*   **Step 2. AI通信のテスト（スモークテスト）**: 空の仮画像とテキストをAPIに送り、Claudeの返答が正しくアプリ側で受け取れるかだけを確認。エラーがあればここで潰す。
*   **Step 3. 画像リサイズと認識機能**: 画面から画像をアップロードし、アプリ内で縮小・送信し、食材が手に入るかを確認。
*   **Step 4. UIモック作成とレシピ生成**: 冷蔵庫の一覧画面と、レシピ提案画面のレイアウト作成。実際にレシピを生成し、在庫が減るかを確認。
*   **Step 5. OGP等のブラッシュアップ**: SNSシェア用の画像生成機能や、細かなデザイン調整を行いMVP完成。

## 7. Claude Code への渡し方

開発を Claude Code に依頼する際は、以下の手順を推奨します。

1.  この `implementation_plan.md` と `task.md` をプロジェクトのルートディレクトリに配置します。
2.  Claude Codeを起動し、以下のプロンプトを投げてください：
    > 「`implementation_plan.md` の設計を理解し、`task.md` の手順に従って開発を進めてください。各ステップが終わるごとに報告と動作確認をお願いします。」
3.  Claude Codeが自動的にファイルの作成・編集・環境構築（`npm install` 等）を進めてくれます。
